---
title: "Sequencer & Prover"
description: "How the sequencer obtains VRF proofs, how production mode keeps the secret key out of op-geth, and what is anchored on L1."
---

The VRF secret key is the only off-chain secret in the randomness protocol. The protocol boundary is intentionally narrow: `op-node` may request proofs, but `op-geth` only receives public proof material and never receives `sk`.

## Components

<CardGroup cols={3}>
  <Card title="op-node" icon="diagram-project">
    Builds `PayloadAttributes` for the next L2 block. When Enshrined VRF is active, it computes the seed and asks a `VRFProver` for `(beta, pi)`.
  </Card>
  <Card title="VRFProver" icon="key">
    A small interface with `Prove(seed)`, `PublicKey()`, and `Close()`. Local mode is for development; TEE mode delegates proof generation over gRPC.
  </Card>
  <Card title="op-geth" icon="microchip">
    Receives payload attributes, validates VRF field lengths, and writes the public key and randomness commitment through system deposit transactions.
  </Card>
</CardGroup>

## Prover modes

<AccordionGroup>
  <Accordion title="TEE mode">
    `TEEVRFProver` connects to `vrf-enclave` over TCP or a Unix socket. The enclave service keeps the secp256k1 private key in its process, returns a 33-byte public key, and produces one ECVRF proof for each 32-byte seed.

    This is the production-oriented mode because it keeps `sk` out of both `op-node` and `op-geth`.
  </Accordion>
  <Accordion title="Local mode">
    `LocalVRFProver` loads a hex-encoded development key into `op-node` memory. It is useful for devnets and tests, but it breaks the unpredictability assumption because the operator can read the key and precompute outputs.
  </Accordion>
  <Accordion title="No prover">
    Verifier nodes do not configure a prover. They use VRF fields extracted from batch data to reconstruct the same deposit transaction that the sequencer executed.
  </Accordion>
</AccordionGroup>

## Key lifecycle

<Steps>
  <Step title="Enclave starts with sealed storage">
    `vrf-enclave` either unseals an existing key from its seal directory or generates a new secp256k1 key and seals it. A deployment must explicitly provide a seal key source: `--seal-key-file`, `VRF_ENCLAVE_SEAL_KEY`, or `--dev-seal`.
  </Step>
  <Step title="op-node fetches the public key">
    `TEEVRFProver` connects to the enclave at startup and caches the compressed 33-byte public key returned by `GetPublicKey`.
  </Step>
  <Step title="L1 anchors the accepted key">
    Governance or deployment tooling writes the accepted key into `SystemConfig.vrfPublicKey()`. `op-node` carries that value into payload attributes, and `op-geth` mirrors it into the L2 predeploy with `setSequencerPublicKey(bytes)`.
  </Step>
  <Step title="One proof per block">
    For each block, `op-node` computes `seed = sha256(uint256(blockNumber) || uint256(nonce))` and calls `Prove(seed)`. The returned `(beta, pi)` becomes part of `PayloadAttributes`.
  </Step>
  <Step title="Graceful shutdown zeroes memory">
    The local prover zeroes its key on `Close()`. The enclave server also zeroes its in-memory key after the gRPC server stops.
  </Step>
</Steps>

## Attestation status

The repository includes a development attestation mode for `vrf-enclave`.

<Warning>
  `--attestation=dev` returns an HMAC-based report. It proves key possession in a local test harness, but it is not a hardware attestation quote and does not prove that the code is running inside SGX, TDX, or SEV-SNP.
</Warning>

Production deployments should replace dev attestation with a platform quote flow that binds `(enclave measurement, vrfPublicKey)` before the key is accepted in L1 `SystemConfig`.

## Failure behavior

<AccordionGroup>
  <Accordion title="Transient prover failure">
    `op-node` retries proof generation using the configured VRF retry policy. If all attempts fail, payload attribute construction fails and block production halts.
  </Accordion>

  <Accordion title="Missing public key">
    If the L1 `SystemConfig` key is empty, `op-geth` skips the public-key sync deposit. Once a 33-byte key is configured, it can mirror the key into the L2 predeploy.
  </Accordion>

  <Accordion title="Malformed proof material">
    `op-geth` requires a 32-byte seed, 32-byte beta, 81-byte proof, and nonce. Invalid lengths fail payload construction instead of producing a block that cannot be replayed.
  </Accordion>
</AccordionGroup>

## What the operator can still do

<AccordionGroup>
  <Accordion title="Stop producing proofs">
    The operator can stop the prover or block the gRPC path. This is a liveness failure; it halts block production after retries rather than silently producing blocks without randomness.
  </Accordion>

  <Accordion title="Use local mode on a devnet">
    Local mode is explicitly insecure for production. It gives the operator the secret key and therefore future-output visibility.
  </Accordion>

  <Accordion title="Censor or reorder transactions">
    The operator is still a sequencer. VRF prevents forged randomness; it does not solve the single-sequencer censorship problem.
  </Accordion>
</AccordionGroup>

## What op-geth cannot do

<Check>Extract or use the VRF secret key.</Check>
<Check>Build a valid post-fork payload without the required VRF proof fields.</Check>
<Check>Call `commitRandomness` as a user transaction; the predeploy accepts only the OP Stack `DEPOSITOR_ACCOUNT`.</Check>
