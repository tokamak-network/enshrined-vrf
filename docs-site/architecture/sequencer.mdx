The VRF secret key (`sk`) is the single piece of off-chain state this chain trusts. Everything hinges on **what is allowed to know `sk`** and **how that is proven**.

## The trust boundary

<CardGroup cols={3}>
  <Card title="Enclave knows sk" icon="lock">
    Generated inside the TEE, sealed to the enclave's measurement. Never leaves in plaintext.
  </Card>
  <Card title="Sequencer operator does NOT" icon="user-slash">
    The operator runs the enclave but cannot read its memory. An attacker with root on the sequencer host still cannot extract `sk`.
  </Card>
  <Card title="L1 knows the public key" icon="scroll">
    `SystemConfig.vrfPublicKey()` on L1 records which `pk` the chain accepts. Anyone can cross-check.
  </Card>
</CardGroup>

## Key lifecycle

<Steps>
  <Step title="Key generation inside the enclave">
    On first startup, the enclave generates a secp256k1 key pair using its hardware RNG. The private key never leaves; the public key is emitted for attestation.
  </Step>
  <Step title="Attestation">
    The enclave produces a remote attestation binding `(enclave_measurement, vrfPublicKey)`. A verifier receiving this attestation can confirm:
    - the code running inside the enclave matches a known-good build
    - the advertised `vrfPublicKey` was generated by that code
  </Step>
  <Step title="Key anchoring on L1">
    A governance transaction on L1 sets `SystemConfig.vrfPublicKey(pk)` after attestation review. This is the canonical record of which key the chain accepts.
  </Step>
  <Step title="Derivation pipeline picks it up">
    `op-node` reads `vrfPublicKey` from `SystemConfig` and includes it in `PayloadAttributes`. Every block builder knows which public key to expect.
  </Step>
  <Step title="Per-block proofs">
    The enclave signs one ECVRF proof per block. `op-geth` embeds it in the commit deposit tx (see [Execution path](/architecture/execution)).
  </Step>
  <Step title="Sealed shutdown">
    On operator-initiated shutdown, the enclave zeroes `sk` in memory before exiting. A restart requires re-attestation against the stored sealed key-file (see `vrf-enclave --seal-key-file`).
  </Step>
</Steps>

## What the operator can still do

<AccordionGroup>
  <Accordion title="Refuse to produce proofs">
    The operator can stall the enclave, causing blocks to miss their VRF commitments. This is a liveness attack, not a correctness attack.

    **Mitigation:** the chain treats missing commitments as a block-invalidity condition. A backup sequencer can be rotated in.
  </Accordion>

  <Accordion title="Equivocate across forks">
    The operator could run two attested enclaves simultaneously and feed proofs to competing chain tips.

    **Mitigation:** `SystemConfig.vrfPublicKey` pins exactly one `pk`. Both enclaves would use the same key; fork choice still serializes them.
  </Accordion>

  <Accordion title="Censor specific games">
    The operator is still a sequencer. They can reorder or omit transactions.

    **Mitigation:** this is the same censorship problem every single-sequencer L2 has. It is not unique to VRF and is addressed by the OP Stack's force-inclusion path, not by enclave design.
  </Accordion>
</AccordionGroup>

## What the operator cannot do

<Check>Extract `sk` and produce proofs outside the enclave.</Check>
<Check>Forge a proof for a seed that was never seen by the enclave — the L1 `VRFVerifier` would catch it during a fault proof.</Check>
<Check>Change `vrfPublicKey` without an L1 governance transaction.</Check>

## Enclave implementation

The enclave source lives in [`vrf-enclave/`](https://github.com/tokamak-network/enshrined-vrf/tree/main/vrf-enclave). Highlights:

- **`vrf-enclave serve`** — production loop, serves proof requests from `op-geth`.
- **`vrf-enclave --seal-key-file`** — persistent keystore bound to the enclave measurement.
- **`vrf-enclave --attestation`** — emits the attestation blob for governance review.
- **`vrf-prove`** — standalone CLI for producing one-off proofs in test setups.
