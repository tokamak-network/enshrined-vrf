---
title: "Fault proof"
description: "How VRF commitments remain replayable in disputes, and what the L1 helper checks today."
---

The fault-proof path ensures that a dishonest sequencer cannot finalize state built from fake randomness. The important property is replayability: verifier nodes and fault-proof programs can reconstruct the same VRF deposit transaction from public data, without access to the VRF secret key.

## Where VRF fits in derivation

<Steps>
  <Step title="The sequencer commits randomness in the block">
    `op-geth` inserts a deposit transaction that calls:

    ```solidity
    EnshrainedVRF.commitRandomness(nonce, seed, beta, pi)
    ```

    The resulting L2 state includes the current block's beta and the historical `(seed, beta, pi)` record.
  </Step>
  <Step title="The batch carries VRF evidence">
    During batch creation, derivation scans deposit transactions for the VRF commit and records `VRFSeed`, `VRFProofBeta`, `VRFProofPi`, `VRFNonce`, and `VRFEnabled`.
  </Step>
  <Step title="Verifier nodes reconstruct attributes">
    Verifier nodes do not have a `VRFProver`. When a batch has `VRFEnabled`, derivation copies the batch's VRF fields back into `PayloadAttributes` so `op-geth` reconstructs the same commit deposit.
  </Step>
  <Step title="Fault-proof execution checks the same transition">
    The dispute game eventually narrows to an execution step. The fault-proof VM re-executes the L2 transition with the same payload attributes and predeploy code.
  </Step>
</Steps>

## L1 helper contract

`VRFVerifier` is a dispute helper, not the normal production path. The current contract exposes:

| Function | Check |
| -------- | ----- |
| `verifySeed(blockNumber, nonce, expectedSeed)` | Recomputes `sha256(abi.encodePacked(blockNumber, nonce))`. |
| `verifyProofStructure(pk, seed, beta, pi)` | Checks public-key length, proof length, scalar bounds, and `proofToHash(pi) == beta`. |
| `proofToHash(pi)` | Extracts gamma from the 81-byte proof and computes the ECVRF proof-to-hash output. |
| `computeSeed(blockNumber, nonce)` | Convenience function for seed reconstruction. |

<Warning>
  `verifyProofStructure` does not perform full elliptic-curve verification. Full ECVRF verification is delegated to fault-proof re-execution, where the L2 ECVRF verify precompile at `0x0101` is available to the program.
</Warning>

## Threat model addressed

<CardGroup cols={2}>
  <Card title="Wrong seed" icon="shield-halved">
    Sequencer commits a seed that does not match the block number and nonce.
    **Caught by:** `verifySeed` or deterministic derivation replay.
  </Card>
  <Card title="Wrong beta for proof" icon="shield-halved">
    Sequencer commits a beta that is not the proof-to-hash value of `pi`.
    **Caught by:** `verifyProofStructure` / `proofToHash`.
  </Card>
  <Card title="Malformed proof" icon="shield-halved">
    Sequencer commits proof material with invalid length or scalar bounds.
    **Caught by:** `verifyProofStructure` and `op-geth` payload validation.
  </Card>
  <Card title="Missing commitment" icon="shield-halved">
    Sequencer tries to build a post-fork block without VRF proof fields.
    **Caught by:** `op-node` attribute construction and `op-geth` payload-building validation.
  </Card>
</CardGroup>

## What remains outside the L1 helper

The helper does not prove every part of the security model on its own:

- **Full ECVRF relation:** checked through fault-proof VM re-execution with the ECVRF precompile path, not by the Solidity helper alone.
- **Secret-key confidentiality:** depends on the prover deployment. TEE-backed mode is the production target; local mode is for development only.
- **Transaction censorship or ordering:** handled by the OP Stack sequencing and force-inclusion model, not by VRF.
- **Attestation review:** accepting a `vrfPublicKey` in L1 `SystemConfig` is a governance/deployment decision.

## Dispute outcome

If a fault proof shows that the proposed output root depends on an invalid VRF transition:

1. The disputed output root is invalidated according to the OP Stack dispute game.
2. The proposer's bond is subject to the dispute game's normal penalties.
3. L2 state rolls back to the last valid output. Game state that depended on the invalid randomness rolls back with it.
