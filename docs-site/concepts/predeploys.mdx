A **predeploy** is a contract deployed at genesis at a fixed address. Enshrined VRF is implemented this way today. Session-account contracts are currently deployed by the demo script and reserved for fixed predeploy addresses once that feature is enshrined.

  `GameHubAccount` is not a predeploy — it is deployed per user by the factory. Its code is identical across all users because it is created via CREATE2 with a fixed init-code hash.
</Info>

## Why reserve fixed addresses

Three reasons.

**Gas.** A predeploy call is a direct `SLOAD`/`SSTORE` on the EVM. An equivalent "user-deployed library" would require a `DELEGATECALL` through an application contract, paying more gas every time.

**UX.** Frontends, wallets, and tooling can hard-code a single address. There's no "which VRF contract is this game using?" question.

**Trust.** Predeploys are upgraded through the same mechanism that upgrades the chain itself. A user trusting the chain implicitly trusts the predeploys.

## Predeploy upgrade path

Enshrined predeploys follow the standard L2 hardfork flow:

1. A proposal changes the predeploy's bytecode at a specific block.
2. Sequencer and challenger nodes adopt the new code.
3. The old storage layout is preserved where possible; breaking changes ship with a migration function.

See [Architecture › Execution](/architecture/execution) for the exact upgrade machinery.

## Addressing convention

The target addresses use the same `0x4200…` prefix as standard OP Stack predeploys. The low-order suffix identifies the specific contract:

| Suffix | Contract             |
| ------ | -------------------- |
| `00F0` | `EnshrainedVRF`      |
| `00A0` | `GameHubFactory`     |
| `00A1` | `SessionKeyManager`  |
| `00A2` | `GameRegistry`       |

Only `EnshrinedVRF` is already wired as an OP Stack predeploy in the current implementation. The session-account addresses are reserved by spec until their enshrinement work lands.
