마지막 Sequencer가 조작할 수 있는가?

Threshold VRF에서 A, B, C 세 명의 Sequencer가 있을 때,
A와 B의 값을 모두 본 C가 난수를 조작할 수 있는가?

마지막 공개자의 우위 (Last Revealer Advantage)
단순한 Threshold VRF에서는 Sequencer들이 partial proof를 순서대로 공유합니다.
마지막으로 공유하는 사람이 불공정한 이점을 가집니다.

대기중...
1

A가 partial_A를 공유

A는 자신의 partial proof를 계산해서 브로드캐스트합니다. A 혼자서는 최종 결과를 계산할 수 없습니다.

2

B가 partial_B를 공유

B도 마찬가지로 공유합니다. A와 B 둘만으로는 최종 결과를 계산할 수 없습니다 (3-of-3 필요).

3

C가 partial_A + partial_B를 수신

C는 이제 A의 partial, B의 partial, 그리고 자신의 sk_C를 모두 보유합니다. C는 자기 값을 공유하기 전에 최종 결과를 계산할 수 있습니다.

⚠

C가 결과를 가장 먼저 알게 됨

C가 계산: Combine(partial_A, partial_B, partial_C) → beta
C는 다른 누구보다 먼저 난수를 알게 되고, 공개할지 보류할지 선택할 수 있습니다.

C가 할 수 있는 것:
• 정직하게 공개 — 블록이 정상 생성됨
• partial_C를 보류 — 블록 생성이 지연되고, 재추첨을 강제함 (1-bit 영향력)
• 선행 거래 (Front-run) — 다른 사람들이 결과를 보기 전에 먼저 행동

이것은 Ethereum RANDAO의 "마지막 validator 우위" 문제와 동일한 구조입니다.
C는 난수 값을 선택할 수 없지만, 먼저 볼 수 있고 참여 여부를 결정할 수 있습니다.
해결: P2P 레이어에서 Commit-Reveal 적용
모든 사람이 커밋하기 전까지 다른 사람의 partial을 볼 수 없게 만듭니다.
이 과정은 Sequencer 간 P2P 네트워크에서 일어나며, 온체인이 아닙니다.

✗ 단순 방식 (문제)

1. A가 partial_A 공유
2. B가 partial_B 공유
3. C가 둘 다 보고 → 결과 계산
4. C가 공개 여부를 결정
→ C가 정보 우위를 가짐

✓ Commit-Reveal (해결)

1. A, B, C 각각 hash(partial) 제출
2. 3개의 hash가 모임
3. A, B, C 각각 partial 공개
4. Combine → 결과
→ 커밋 전에 다른 사람의 값을 볼 수 없음

1

Phase 1 — 커밋 (P2P 네트워크)

각 Sequencer가 자신의 partial을 계산하고 hash만 제출합니다.
A → hash(partial_A + salt)
B → hash(partial_B + salt)
C → hash(partial_C + salt)
모두가 hash를 볼 수 있지만, hash에서 실제 partial 값을 역산할 수 없습니다.

🔒

3개의 커밋이 모임 — 잠김

이제 아무도 자기 값을 바꿀 수 없습니다. hash가 이미 기록되었습니다.
C는 A나 B의 실제 partial을 보기 전에 이미 커밋했습니다.

2

Phase 2 — 공개 (P2P 네트워크)

각 Sequencer가 실제 partial + salt를 공개합니다.
P2P 네트워크가 공개된 값이 커밋한 hash와 일치하는지 검증합니다.
hash(partial_A + salt) == committed_hash_A? ✓

?

C가 A, B의 공개를 먼저 보고 조작할 수 있는가?

네, 이 시점에서 C는 최종 결과를 계산할 수 있습니다. 하지만 C는 Phase 1에서 이미 커밋했습니다.
C에게 남은 유일한 선택은 공개를 거부하는 것(보류 공격)뿐입니다.

✓

보류 공격 → VSS로 무력화

C가 공개를 거부하면, A와 B가 DKG 시 공유된 commitment을 이용해 C의 키 조각을 복원할 수 있습니다 (Verifiable Secret Sharing).
블록 생성은 정상 진행됩니다. C는 슬래싱됩니다.

모든 공격이 차단됨:
• 결과를 먼저 보는 것? — 불가능. Phase 1에서는 hash만 공개됨
• 다른 사람 값을 보고 바꾸는 것? — 불가능. 이미 hash로 커밋됨
• 공개를 거부하는 것? — VSS가 share를 복원하고, C는 슬래싱됨

이 모든 commit-reveal 과정은 Sequencer 간 P2P 네트워크에서 일어나며, 온체인이 아닙니다.
사용자가 보는 것
위의 모든 과정은 블록이 생성되기 전에 일어납니다.
사용자 경험은 전혀 바뀌지 않습니다.

뒷단에서 일어나는 일 (블록 생성 전)

1

TX 순서 합의

Sequencer들이 BFT로 트랜잭션 순서에 합의합니다. 순서가 잠깁니다.

2

Partial proof 커밋 (P2P)

각 Sequencer가 hash(partial)을 제출합니다. 실제 값은 아무도 볼 수 없습니다.

3

Partial proof 공개 (P2P)

각 Sequencer가 partial을 공개합니다. 합쳐서 최종 beta가 됩니다.

4

블록 생성

commitRandomness(nonce, beta, proof)가 deposit TX로 포함됩니다.

▼

사용자의 트랜잭션 (변경 없음)

여전히 1개의 트랜잭션, 여전히 같은 블록 안에서.

uint256 random = VRF.getRandomness();

Commit-reveal 오버헤드는 블록 생성 시간에 약간의 레이턴시(~100ms)를 추가할 뿐,
사용자 트랜잭션 수가 늘어나는 것이 아닙니다.

요약:

Q: 마지막 Sequencer가 난수를 조작할 수 있는가?
A: 아니오. Commit-reveal로 커밋 전에 다른 사람의 partial을 볼 수 없습니다.

Q: 마지막 Sequencer가 참여를 거부할 수 있는가?
A: 네, 하지만 VSS로 다른 참여자들이 빠진 share를 복원합니다. 블록은 정상 생성되고 공격자는 슬래싱됩니다.

Q: 사용자에게 추가 트랜잭션이 필요한가?
A: 아니오. Commit-reveal은 블록 생성 전 Sequencer 간 P2P 네트워크에서 일어납니다. 사용자는 여전히 getRandomness() 한 번으로 같은 트랜잭션에서 결과를 받습니다.
