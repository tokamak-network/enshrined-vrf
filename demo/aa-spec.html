AA IMPLEMENTATION SPEC & RATIONALE

최소 AA — 세션 키 UX만을 위한 컨트랙트 레이어 설계

enshrined-vrf의 AA는 ERC-4337도 EIP-7702도 RIP-7560도 아닌 컨트랙트 레이어의 커스텀 구현입니다. 기존 AA 스택을 그대로 채택하지 않은 데에는 이유가 있고, 빠진 기능에도 이유가 있습니다. 이 문서는 스펙(WHAT)과 근거(WHY)를 좌우로 배치해 각 결정을 그대로 드러냅니다.

01

어느 레이어에 서는가

AA는 여러 레이어에서 구현 가능합니다. 각 레이어는 서로 다른 비용과 유연성을 요구합니다. 우리는 컨트랙트 레이어를 선택했지만, ERC-4337의 bundler 네트워크는 채택하지 않았습니다.

02

무엇을 이루고, 무엇을 포기했는가

목표와 비목표는 같은 무게로 중요합니다. "무엇을 하지 않기로 했다"가 설계를 단순하게 만듭니다.

Primary Goals (Phase 1)

  • 입금 1서명, 게임 0서명. 모든 다른 요구사항은 이 목표를 해치지 않는 범위에서.
  • 세션 키 scope 강제. 유출되어도 피해가 제한적이어야 함 (game, cap, expiry, selectors).
  • 인출은 절대 세션 키로 불가능. 최악의 시나리오에서도 자금 유출 차단.
  • 누구나 tx 제출 가능. 검열 저항, 전용 bundler 네트워크 불필요.
  • 결정적 Hub 주소. EOA 주소만 알면 Hub 주소 계산 가능 → pre-onboarding UX.
  • OP Stack 변경 없음. 4개 Solidity 컨트랙트로 완결. 포크 추가 유지보수 비용 없음.

Non-goals (의식적으로 뺌)

  • ERC-4337 호환성. EntryPoint/UserOp 포맷 채택하면 bundler 인프라 필요 + 게이밍 UX에 과잉.
  • EIP-7702 통합. EOA → Hub delegation은 매력적이나 Phase 3 이후. 지금은 별도 Hub 주소 모델.
  • Paymaster / gas sponsorship. Phase 5 옵션. 현재는 Relayer가 자기 가스로 내고 오프체인 비즈니스 모델로 회수.
  • Passkey / WebAuthn signers. MVP는 ECDSA만. P256 precompile 있으니 후속 확장 가능.
  • 소셜 복구 / 멀티시그. 게임 UX 범위 밖. Hub 소유자는 단일 EOA.
  • Cross-game asset transfers. 자금은 게임별 balances[game]에 earmark, 게임 간 이동 불가.

04

컨트랙트 4종 — 스펙 & 근거

각 컨트랙트마다 무엇을 하는지를 왼쪽에, 왜 이 모양인지를 오른쪽에 나란히 배치합니다.

GameHubAccount PER-USER · CREATE2
contracts/src/aa/GameHubAccount.sol
SPEC

사용자당 1개의 스마트 계정

자금을 게임별 슬롯으로 보관, 세션 키 서명을 검증하고 scope 안에서 게임을 대신 호출.

  • immutables: owner, factory, sessionKeys, registry
  • storage: balances[game], sessionNonce[sessionKey]
  • executeAsOwner(game, value, data) — EOA만 호출 가능, 임의 게임 호출
  • executeAsSession(skey, game, value, data, nonce, sig) — 누구나 제출 가능, Hub 내부에서 sig·scope 검증
  • deposit(game) payable / withdraw(game, amount, to) — 입출금
  • registerSession / refillSession / revokeSession — 세션 라이프사이클
  • modifier: onlyOwnerOrFactory (deposit · register · refill)
WHY

왜 이 모양인가

  • 사용자당 1개 Hub (게임당 X) — 키 관리를 단순화하고 주소를 고정. 여러 게임을 오가도 자금 창고는 하나.
  • Factory modifier 허용 — 온보딩을 1 signature로 원자화. Factory 없으면 deploy·deposit·register가 3 tx.
  • Owner/Session 경로 타입으로 분리 — executeAsOwner는 scope 무시, executeAsSession은 scope 강제. 권한 차이가 함수 시그니처에 그대로 드러남.
  • 세션 키별 독립 nonce — 여러 세션이 동시에 parallel tx 제출 가능 (RIP-7712 2D nonce 유사 효과).
  • Per-game balance slot — CoinFlip에 베팅한 돈이 Plinko로 새지 않음. 한 게임 버그가 전체 자금을 위협하지 않음.
GameHubFactory PREDEPLOY · 0x42…A0
contracts/src/aa/GameHubFactory.sol
SPEC

결정적 Hub 배포 + 원자적 온보딩

  • hubOf(owner) view returns (address) — CREATE2 주소 계산 (salt = bytes32(uint256(uint160(owner))))
  • depositAndRegister(sessionKey, scope) payable — Hub 미존재 시 배포 → deposit forward → registerSession forward
  • depositAndRefill(sessionKey, game, addedCap, newExpiry) payable — 원자적 리필
WHY

원자성과 결정성의 힘

  • CREATE2 salt = owner — owner만 알면 Hub 주소 오프체인 계산 가능. 게임 UI가 "Hub 주소가 이것입니다"를 먼저 보여줄 수 있음.
  • 원자적 deploy+deposit+register — 중간 단계에서 tx가 실패하면 전체 롤백. 사용자가 "Hub는 있는데 세션 등록 실패" 같은 이상 상태에 놓이지 않음.
  • Factory가 Hub에 forward 호출 — Hub 내부에서 msg.sender = Factory가 되어 onlyOwnerOrFactory 통과. Factory가 권한 uplift 경로.
SessionKeyManager PREDEPLOY · 0x42…A1
contracts/src/aa/SessionKeyManager.sol
SPEC

모든 Hub이 공유하는 scope 레지스트리

  • storage: _scopes[hub][sessionKey] = Scope, _active[hub][sessionKey] = bool
  • Scope = { gameAddr, spendingCap, expiry, bytes4[] selectors }
  • register(sessionKey, scope) — msg.sender가 Hub, 해당 Hub의 scope에 기록
  • canCall(hub, skey, game, selector, value) view — 4중 체크 (active · gameAddr · expiry · cap · selector)
  • consume(sessionKey, amount) — 성공한 exec 후 Hub이 cap 차감
  • refill(skey, addedCap, newExpiry) / revoke(skey)
WHY

Hub에 내장하지 않은 이유

  • 공유 레지스트리 = 프리디플로이 승격 후보. 모든 Hub이 같은 SKM을 바라보게 하면 Phase 4에서 프리디플로이 한 곳에 하드코딩하면 됨.
  • "msg.sender가 Hub" 불변식. EOA가 SKM을 직접 호출해도 자기 주소 slot에만 기록 → 실제 Hub의 scope는 건드릴 수 없음. 별도 접근 제어 불필요.
  • canCall은 view, consume은 mutation. 의사결정권자는 Hub. SKM은 "스코프 저장소 + 검증 유틸"로 역할 국한.
  • Scope에 selectors 포함. 같은 게임 안에서도 함수 단위 제한 (예: flip 허용, emergencyWithdraw 금지). 게임 업그레이드 시 세션 무효화 가능.
GameRegistry PREDEPLOY · 0x42…A2
contracts/src/aa/GameRegistry.sol
SPEC

게임 컨트랙트 화이트리스트

  • register() — msg.sender를 게임으로 등록, tx.origin을 등록자로 기록
  • isRegistered(game) view returns (bool)
  • deregister(game) — 등록 시점 tx.origin만 가능
  • 게임 컨트랙트는 자기 constructor에서 registry.register() 호출
WHY

자기 등록 패턴의 강점

  • 등록 권한이 컨트랙트 코드 자체. EOA가 임의로 "악성 컨트랙트"를 게임으로 spoof 등록 불가. 게임 컨트랙트의 constructor 코드가 실제로 register()를 부르는지 검증 가능.
  • 세션 키의 call target 제한. 등록 안 된 컨트랙트로는 세션 키가 자금을 옮길 수 없음 → rug pull 방지.
  • 별도 컨트랙트로 분리. SKM과 마찬가지 이유 — 모든 Hub이 공유하는 global 레지스트리 → 프리디플로이 승격 용이.
05

서명 스킴

세션 키가 서명하는 digest의 구성과, 왜 EIP-712가 아닌 \x19 prefix를 쓰는지.

// GameHubAccount.sol :: _digest() inner = keccak256(abi.encode( address(this), // hub 주소 — 다른 Hub로 replay 불가 block.chainid, // chain 바인딩 — 크로스체인 replay 불가 sessionKey, // 어느 세션 키인지 game, // 어느 게임을 호출하는지 value, // 얼마 쓰는지 keccak256(data), // 어떤 calldata인지 (가변 → 고정해시) nonce // sessionNonce[skey] — replay 방지 )); digest = keccak256(abi.encodePacked( "\x19Ethereum Signed Message:\n32", inner )); signer = ecrecover(digest, v, r, s); require(signer == sessionKey);
SPEC

digest에 들어가는 7개 필드

  • address(this), chainid: replay 경계 정의
  • sessionKey, game, value: 무엇을 누구에게 얼마로 할 것인가
  • keccak(data): calldata 바인딩 (길이 가변 → 고정)
  • nonce: 세션 키별 단조 증가, 동일 sig 두 번 불가
  • 서명: 표준 ECDSA(secp256k1), 65 bytes (r, s, v)
  • 검증: ecrecover 1회 + SKM.canCall 1회
WHY

각 선택의 근거

  • EIP-712 대신 \x19Ethereum Signed Message: 게임 sig는 짧고 빈번. Structured data의 도메인 separator·type string 오버헤드 대비 이득 미미. 웹3 라이브러리의 signMessage API로 바로 서명 가능 → SDK가 단순. EIP-712 전환은 나중에 쉬움.
  • secp256k1만 (WebAuthn X): OP Stack 기본 내장. P256 precompile은 0x0100에 이미 있어 passkey는 Phase 후반 확장 후보.
  • address(this) + chainid: cross-Hub / cross-chain replay 원천 차단. 사용자가 테스트넷에서 실험한 sig를 누가 메인넷에 재제출해도 ecrecover는 다른 signer를 반환.
  • 세션 키별 독립 nonce: 여러 세션이 병렬 submit 해도 간섭 없음. 전역 nonce였다면 throughput 병목.
  • Hub이 ecrecover 직접 호출: EntryPoint 스타일 indirection 없음. "Hub = validator" 원칙에 충실, 오버헤드 최소.
06

신뢰 모델 & 불변식

시스템의 안정성은 몇 개의 명시적 불변식 위에 서 있습니다. 각 불변식이 어떤 공격을 막는지도 함께.

INVARIANT 1
"Hub만이 자기 자신의 세션 키를 등록할 수 있다"
SKM.register에서 msg.sender를 Hub 주소로 기록. EOA가 SKM을 직접 호출해도 자기 주소 slot에만 기록돼 Hub scope는 오염 불가.
INVARIANT 2
"세션 키는 withdraw 경로에 진입할 수 없다"
Hub.withdraw는 msg.sender == owner 체크. executeAsSession은 SKM.canCall에서 gameAddr 일치 강제 → Hub 자신을 target으로 부를 수 없음.
INVARIANT 3
"세션 키는 GameRegistry에 등록된 컨트랙트만 호출할 수 있다"
Scope의 gameAddr는 Hub.deposit에서 registry.isRegistered(game) 통과한 주소로만 설정됨. 악성 컨트랙트는 등록 못 하므로 targeting 불가.
INVARIANT 4
"한 게임에 earmark된 자금은 다른 게임으로 이동할 수 없다"
balances[game] slot 분리. executeAsSession은 balances[game] -= value만 차감, 다른 slot 건드리지 않음.
INVARIANT 5
"같은 서명을 두 번 제출할 수 없다"
sessionNonce[skey]가 단조 증가. digest에 nonce 포함 + Hub이 nonce == sessionNonce[skey] 검증 + 성공 시 increment.
INVARIANT 6
"Hub 주소는 owner 주소로부터 결정적으로 유도된다"
Factory가 CREATE2, salt = bytes32(owner). 사용자당 1개의 Hub만 존재 가능. 오프체인 UI가 EOA만 알면 Hub 주소 계산.

공격 시나리오별 방어

⚠세션 키 유출
피해는 (spendingCap, expiry, gameAddr, selectors)에 한정. 인출·다른 게임 호출 불가. EOA는 revokeSession으로 즉시 무효화.
⚠Hub tx replay
sessionNonce가 increment → 동일 sig + 동일 calldata 재제출 시 BadNonce revert.
⚠Cross-chain replay
digest에 block.chainid 포함 → 다른 chain에서 ecrecover가 다른 signer 반환.
⚠Cross-Hub replay
digest에 address(this) 포함 → 다른 Hub로 동일 sig 재사용 불가.
⚠악성 컨트랙트로 자금 이동
GameRegistry 미등록 컨트랙트는 scope.gameAddr가 될 수 없음. 등록은 컨트랙트 생성자 자기 호출만 가능.
⚠과도한 지출
SKM.consume이 cap 차감, underflow 시 solidity 0.8+ revert.
⚠기간 초과 사용
canCall에서 s.expiry ≤ block.timestamp 체크.
⚠Relayer 검열
아무나 executeAsSession 제출 가능 — 사용자 자신의 지갑, 다른 relayer, 게임 백엔드 어느 경로든. 단일 bundler 의존성 없음.
07

표준 AA와의 비교

같은 "계정 추상화"라도 채택한 레이어·인프라 요구·호환성이 제각각입니다.

차원 ERC-4337 EIP-7702 RIP-7560 (Native AA) enshrined-vrf AA ★
Entry point 공유 EntryPoint EOA 자체 (delegated code) 프로토콜 (tx 타입) Hub 자체가 entry
Tx 포맷 UserOperation 일반 tx + authorization_list 신규 AA tx type 일반 tx, data = executeAsSession(sig, ...)
제출자 (submitter) 전용 Bundler 네트워크 누구나 Sequencer 누구나 (릴레이어/지갑/백엔드)
Signer 추상화 Account가 validateUserOp 구현 Delegated code가 validate Account가 validate Hub이 ecrecover + SKM scope
Gas 후원 Paymaster 표준화 직접 X, 별도 설계 필요 Paymaster 네이티브 현재 없음, Phase 5 옵션
구현 레이어 컨트랙트 실행 레이어 (EL) 합의/실행 레이어 컨트랙트 (프리디플로이 후보)
배포 복잡도 EntryPoint + Bundler 인프라 EL 포크 EL + CL 포크 4개 Solidity 컨트랙트
UserOp / sig mempool 별도 Alt-mempool 기존 mempool 기존 mempool 기존 mempool
Why not ERC-4337? UserOp 포맷·EntryPoint·Bundler 네트워크가 주는 유연성은 컸지만, 게임 세션 키 UX라는 단일 요구에 비해 과잉. 우리는 executeAsSession 하나에 sig를 calldata로 담아 기존 tx mempool로 흘려보내는 쪽을 택했습니다. 훗날 ERC-4337 host에도 올리고 싶다면 Hub이 IAccount 구현 추가하면 됩니다.
08

주요 설계 결정

각 결정에는 버린 대안이 있습니다. 그 대안을 명시적으로 기록해두는 게 설계의 일관성을 유지하는 유일한 방법입니다.

선택한 것 버린 대안 근거
사용자당 1개 Hub 게임당 1개 Hub (Hub × games) 게임이 늘 때마다 Hub가 늘면 키 관리 복잡. 하나의 Hub에 per-game balance slot으로 격리하면 충분.
SKM을 Hub에 분리 Hub 내부 mapping(skey => Scope) 프리디플로이 승격 용이. 모든 Hub이 공유하는 global 레지스트리 → Phase 4에서 genesis에 하드코딩.
GameRegistry 분리 Hub마다 자체 화이트리스트 게임 등록을 탈중앙화. 게임 배포자가 자기 생성자에서 register() 호출하면 끝. 중앙 권한자 없음.
CREATE2 salt = owner 주소 keccak(owner, seed) with seed owner만 알면 Hub 주소 계산 가능 → pre-onboarding UI, 주소 예측 가능성 확보.
ECDSA(secp256k1)만 WebAuthn/passkey (secp256r1) MVP 단순성. OP Stack에 P256 precompile 이미 있으므로 Phase 후반에 multi-sig-scheme으로 확장 가능.
\x19Ethereum Signed Message digest EIP-712 structured data 짧고 빈번한 게임 sig에 대한 복잡도 vs. 이득 판단. 지갑/브라우저 기본 지원.
누구나 tx 제출 가능 화이트리스트된 Bundler만 검열 저항, bundler 네트워크 운영 부담 없음. Sig 검증은 Hub이 하므로 제출자는 trust boundary 밖.
세션 키 = 게임 1개 세션 키 = 복수 게임 scope Scope 구조 단순화. 게임 간 권한 격리 명시적. 복수 게임 쓰려면 게임당 세션 키 발급.
세션 키는 인출 불가 cap 내에서 인출 허용 "세션 키 = 소비 권한, 인출은 main wallet" 원칙. 유출 시 자금 완전 보호.
onlyOwnerOrFactory onlyOwner만 Factory가 원자적 온보딩 수행 가능하게. Factory는 trusted predeploy이므로 권한 uplift 경로로 허용.
Per-game balance slot 단일 pool balance 한 게임 버그가 다른 게임 자금 위협하지 않음. 세션 키 scope와 1:1 대응.
canCall은 view, consume은 mutation 단일 consumeIfAllowed 의사결정 주체를 Hub으로 명시. Hub이 판단 후 상태 변경 명령. SKM은 저장소 역할.
Paymaster 없음 (Phase 1) 초기부터 Paymaster 포함 MVP scope 축소. 현재 Relayer가 자기 가스 지불 후 오프체인 비즈니스 모델로 회수.

단계적 로드맵

프로토콜 enshrining은 설계가 안정화된 뒤에. 각 phase는 이전 phase의 결과로 결정이 바뀔 수 있음을 전제합니다.

PHASE 1 · DONE

Solidity 프로토타입

contracts/src/aa/ — 4개 컨트랙트 + Foundry 테스트
4개 컨트랙트(GameHubAccount, GameHubFactory, SessionKeyManager, GameRegistry) 구현 완료. contracts/test/SessionAccounts.t.sol이 3가지 lifecycle(deposit / play / withdraw) 커버.
컨트랙트로 먼저 만드는 이유: 프로토콜 포크에 고정하기 전에 설계 검증. 잘못된 설계를 genesis에 하드코딩하면 되돌리기 어려움.
PHASE 2 · NOW

브라우저 E2E 데모

demo/session-play.html — 지갑 연결, 실제 체인 상호작용

EIP-7702 통합

"Hub 주소 따로 관리" 멘탈 모델을 제거해서 기존 지갑 사용자의 온보딩 장벽 낮춤.

설계가 안정된 뒤에만 — Phase 1-3에서 검증된 인터페이스만 영구 고정. 변경 비용이 큰 결정이므로 마지막에.
